03 · Fluxo de uma pergunta

Trace passo-a-passo de POST /query {"question":"Qual foi a receita de Search entre 2024-01-01 e 2024-01-31?"}:

  1. HTTP recebe em query_analytics (app/main.py:198). Valida via QueryRequest (Pydantic). Se faltar thread_id, gera UUID.
  2. Invoca grafo persistente via invoke_analytics_graph com thread_id → carrega histórico do MemorySaver.
  3. preprocess_node chama build_router_decision(question). Detecta:
    • intent = channel_performance (tokens receita)
    • traffic_source = Search (alias)
    • start_date / end_date parseadas via YYYY-MM-DD
    • nenhuma clarificação necessária → não short-circuita
    Injeta HumanMessage no estado e roteia para agent.
  4. agent_node monta input: system prompt (build_conversation_system_prompt) + router guidance + histórico do turn. Chama LLM com tools bound (tool_choice="auto").
  5. LLM retorna AIMessage com tool_calls=[{name: "channel_performance_analyzer", args: {traffic_source: "Search", start_date: "2024-01-01", end_date: "2024-01-31"}}].
  6. tool_executor_node resolve a tool em tools_by_name, valida args via ChannelPerformanceInput (Pydantic), executa channel_performance_analyzer que dispara a SQL parametrizada no BigQuery.
  7. BQ retorna rows → empacota em ToolMessage com content JSON + artifact tipado.
  8. agent_node novamente recebe o ToolMessage, sintetiza resposta final em pt-BR com leitura de negócio, sem tool_calls desta vez → final_answer.
  9. HTTP responde com QueryResponse: answer, tools_used, metadata (thread_id, debug se ativo).

Detalhe não óbvio: o guard de iteração

O agent_node conta quantas AIMessage com tool_calls existem no turn atual. Se ≥ _MAX_AGENT_ITERATIONS (3), corta com mensagem temporária e termina. Sem isso, um LLM defeituoso poderia loopar agent → tool_executor → agent → tool_executor infinitamente, queimando tokens e dinheiro.

04 · LangGraph fundamentals

Conceito em 1 parágrafo

LangGraph é um framework de orquestração de agentes baseado em grafo de estado explícito. Diferente do AgentExecutor da LangChain (que esconde o loop ReAct dentro de uma classe), o LangGraph te força a declarar nodes, edges e reducers. Vantagem: você controla short-circuits, debugging, guards de iteração, persistência e ramificação condicional. Custo: mais boilerplate inicial.

Componentes essenciais

ComponentePapel
StateGraph(StateType)Container do grafo, parametrizado pelo TypedDict do estado
NodeFunção pura (state) → Command | dict que muta o estado
EdgeTransição entre nodes; pode ser estática (add_edge) ou retornada via Command(goto=...)
Annotated[list, add_messages]Reducer — diz ao LangGraph que o campo messages deve ser concatenado em vez de sobrescrito
Command(update=..., goto=...)Retorno de node combinando atualização de estado + decisão de roteamento
checkpointerPersiste estado por thread_id; aqui usa MemorySaver in-memory

Note que preprocess e agent usam Command(goto=...) para decidir o próximo node dinamicamente, então não precisam de add_edge estático. Já tool_executor → agent é estático porque sempre volta.

Por que não existe add_edge(..., END) explícito?

O destino "fim do grafo" é decidido em runtime pelo próprio node, via Command(goto="__end__"). A string "__end__" é o sentinel equivalente à constante END de langgraph.graph. Quando node retorna Command com goto, o edge estático fica redundante — funciona como if/else em código normal: ambos os caminhos existem no grafo compilado, só um executa por invocação. Para o Studio desenhar a aresta tracejada, declare o retorno como Command[Literal["agent", "__end__"]] ou use add_conditional_edges.

Pegadinha de entrevista

Por que LangGraph em vez de create_react_agent ou AgentExecutor? Resposta: controle explícito do loop, possibilidade de injetar um preprocess determinístico antes do LLM (economizando tokens em short-circuits), guard de iteração, observabilidade dos nodes individualmente, e mensagens de sistema condicionais por intent (follow-up estratégico vs diagnóstico).

O que é ReAct e onde aparece

ReAct = Reasoning + Acting. LLM alterna Thought (raciocínio) → Action (chama tool com args) → Observation (lê resultado) → repete até resposta final. Hoje "ReAct agent" virou sinônimo do loop LLM → tool → LLM → tool → ... → final, com tool calls em JSON estruturado em vez de texto Thought/Action.

Onde aparece em produção: Claude Code, Cursor, Codex/Copilot Workspace, Aider, Cline, ChatGPT (Code Interpreter/Browse), Claude com tools/MCP, Devin, OpenHands — todos rodam ReAct no núcleo. Diferenças entre produtos não estão no padrão, mas em tool set, gestão de contexto, planning opcional antes do loop e orquestração multi-agente. Variantes acadêmicas (Reflexion, Tree of Thoughts, LATS) existem mas produção comercial usa ReAct puro ~95% das vezes — simples ganha.

Senioridade na escolha: não é "fiz short-circuit funcionar", é trade-off consciente. Júnior pega create_react_agent porque é o que o tutorial mostra; sênior abre mão da conveniência por controle e justifica.

05 · Tool Calling

Conceito em 1 parágrafo

Tool calling (function calling) é o protocolo onde o LLM não executa código; em vez disso, ele retorna um JSON estruturado descrevendo qual função chamar e com quais argumentos. Você executa, retorna o resultado, e o LLM sintetiza. Isso troca alucinação por auditoria: SQL é fixa, validada, parametrizada — o LLM só escolhe parâmetros dentro de um schema declarado.

Ciclo neste projeto

  1. App registra tools com StructuredTool.from_function e schema Pydantic.
  2. App chama model.bind_tools(tools, tool_choice="auto") → LangChain converte o schema Pydantic em JSON schema OpenAI/Anthropic.
  3. LLM recebe a pergunta + descrições das tools, devolve AIMessage.tool_calls.
  4. tool_executor_node resolve por name, valida args com Pydantic, executa, devolve ToolMessage.
  5. LLM agora vê o resultado e produz AIMessage sem tool_calls = resposta final.

06 · Router determinístico

Esclarecimento: o router não é um node do grafo

Router é uma função pura build_router_decision(question: str) → RouterDecision chamada dentro do preprocess_node. Não existe um node "router" no StateGraph. O preprocess_node apenas: (1) pega a pergunta do state, (2) chama build_router_decision, (3) decide Command(goto=...) conforme o resultado. Determinístico de ponta a ponta — sem LLM dentro.

Por que existir fora do LLM

O router é um classificador regex/token-set em Python puro que roda antes do LLM. Custo: zero tokens. Latência: microssegundos. Cobre:

  • Classificação de intent
  • Normalização de datas (5 formatos)
  • Detecção de traffic_source com aliases
  • Guardrails de schema (recusa CAC/ROAS, recusa "por campanha")
  • Decisão de short-circuit (clarificação ou recusa) sem nunca acordar o LLM

Intents possíveis

Trade-off declarado: regex é frágil para PT livre — variações como "qual o trafego dos canais" sem acento podem escapar tokens. A normalização Unicode (NFKD + strip de combining chars) cobre acentos, mas não cobre erros de digitação ou sinônimos não previstos. Custo de adicionar LLM-based router: tokens em toda requisição válida. Decisão: aceitar fragilidade controlada por testes (tests/unit/test_router.py).

Pseudocódigo do classificador

def build_router_decision(question: str) -> RouterDecision:
    norm = _normalize(question)              # NFKD + lower + strip acentos
    tokens = set(norm.split())

    if not norm:
        return RouterDecision(intent="empty", short_circuit=True, ...)

    if tokens & UNSUPPORTED_METRIC_TOKENS:    # CAC, ROAS, CTR, ...
        return RouterDecision(intent="out_of_scope", short_circuit=True, ...)

    start, end = _parse_dates(norm)           # regex (5 formatos)
    source = _detect_traffic_source(norm)     # dict de aliases

    if tokens & REVENUE_TOKENS:
        intent = "channel_performance"
    elif tokens & VOLUME_TOKENS:
        intent = "traffic_volume"
    elif tokens & STRATEGY_TOKENS and has_prev_context:
        intent = "strategy_follow_up"
    else:
        intent = "ambiguous_analytics"

    if intent in ("traffic_volume","channel_performance") and not (start and end):
        return RouterDecision(intent=intent, short_circuit=True,
                              response_message=CLARIFY_DATES_MESSAGE)

    return RouterDecision(intent=intent, traffic_source=source,
                          start_date=start, end_date=end, short_circuit=False)

Pipeline = if/else + regex + lookups em frozenset. Determinístico, O(n) no tamanho da pergunta, microssegundos.

Resposta-modelo para "por que não usou LLM router?"

  1. Escopo fechado — case define 3 tabelas e 2 perguntas-tipo. Espaço de intents cabe em regex.
  2. Critério "fora de escopo" — case avalia recusa explícita. Regex recusa CAC/ROAS com 100% reliability.
  3. Custo previsível — agent já paga LLM. Router LLM dobra custo sem ganho aqui.
  4. Testabilidade — pytest cobre router. LLM router exigiria eval suite, fora do MVP.

Em domínio aberto, inverteria. Híbrido = próximo passo: regex pega óbvios, LLM resolve ambíguos.

Como a continuidade funciona

08 · SQL & BigQuery deep-dive

Query principal — channel_performance

SELECT
    COALESCE(u.traffic_source, 'Unknown') AS traffic_source,    -- (1)(2)
    COUNT(DISTINCT o.order_id) AS total_orders,                  -- (3)
    COALESCE(ROUND(SUM(CAST(oi.sale_price AS NUMERIC)), 2), 0)   -- (4)(5)
        AS total_revenue
FROM `bigquery-public-data.thelook_ecommerce.users` u
INNER JOIN `bigquery-public-data.thelook_ecommerce.orders` o    -- (6)
    ON u.id = o.user_id
INNER JOIN `bigquery-public-data.thelook_ecommerce.order_items` oi
    ON o.order_id = oi.order_id
WHERE DATE(o.created_at) BETWEEN @start_date AND @end_date       -- (7)
    AND o.status = 'Complete'                                    -- (8)
    AND (
            @traffic_source IS NULL                              -- (9)
            OR LOWER(COALESCE(u.traffic_source, 'Unknown'))
               = LOWER(@traffic_source)
    )
GROUP BY COALESCE(u.traffic_source, 'Unknown')
ORDER BY total_revenue DESC, total_orders DESC                   -- (10)

app/tools/channel_performance_analyzer.py:14

1COALESCE traffic_source — NULL vira 'Unknown' para evitar perda silenciosa de linhas no GROUP BY. Sem isso, NULLs viram um bucket implícito difícil de auditar.

2Alias u., o., oi. — disambiguação obrigatória após JOINs com colunas homônimas.

3COUNT(DISTINCT order_id) — crítico. order_items tem N linhas por pedido (1 por item). Sem DISTINCT, um pedido com 3 itens conta como 3 pedidos. Esta linha é o que separa um engenheiro de dados sênior de um júnior na entrevista.

4CAST AS NUMERIC — sale_price está em FLOAT64. Somar floats acumula erro binário (0.1 + 0.2 ≠ 0.3). Em receita, isso é inaceitável. NUMERIC é decimal com precisão fixa.

5ROUND(..., 2) — apresentação em centavos, mas depois do SUM, não antes (round em cada linha enviesa o total).

6INNER JOIN — descarta usuários sem pedido e pedidos sem itens. Correto para "receita por canal": só usuários que compraram contam. Para conversion rate seria LEFT JOIN (precisa do denominador completo).

7BETWEEN inclusivo — datas usam fronteira de dia. Filtra por created_at do pedido, não do usuário (receita realizada no período, não cohort).

8status='Complete' — exclui Cancelled/Returned/Processing. Sem isso, "receita" infla com pedidos que nunca viraram dinheiro real.

9Filtro condicional — quando o usuário não especifica canal, @traffic_source IS NULL deixa todos passarem. LOWER(...) dos dois lados garante case-insensitive.

10 · Pydantic essencial

Por que Pydantic em vez de dataclass

  • Validação em runtime — dataclass não verifica tipos, só armazena.
  • JSON Schema gratuito — model.model_json_schema(). É exatamente isso que o LangChain envia para o LLM no binding de tools.
  • Coerção controlada — string ISO vira date automaticamente via field_validator.
  • Serialização — model_dump(mode="json") resolve Decimal, date, UUID.
  • extra="forbid" — falha em campo extra no input, útil em contrato HTTP.

13 · Decisões a justificar

15 trade-offs que o entrevistador vai sondar. Cada um com TL;DR.

  1. LangGraph em vez de AgentExecutor: controle explícito do loop + guard de 3 iterações + short-circuit no preprocess + nodes inspecionáveis para debug.
  2. Router determinístico antes do LLM: custo (regex grátis), latência (μs), determinismo, guardrails consistentes mesmo se o LLM tiver fallback.
  3. Tools fixas em vez de LLM-gera-SQL: anti-injection, custo BQ previsível, auditoria de query, testes determinísticos.
  4. Duas tools separadas (volume vs performance): descrições mais precisas levam o LLM a escolhas melhores; menos prompt engineering ad-hoc.
  5. tool_choice="auto": permite o LLM responder follow-up estratégico/diagnóstico sem nova query quando contexto já basta.
  6. MemorySaver in-memory: escopo MVP; documentado como limitação; troca direta por RedisSaver em produção.
  7. FastAPI: tipagem nativa + Pydantic + OpenAPI auto. Flask precisaria de Marshmallow/apispec.
  8. temperature=0: respostas analíticas determinísticas; anti-alucinação em síntese de números.
  9. pt-BR + leitura de negócio: público real é mídia BR, não desenvolvedor. Produto não é query runner.
  10. CLI consome a própria API: mesmo contrato de cliente real; descobriria bug de serialização antes do front existir.
  11. Fallback OpenAI ↔ Anthropic: resiliência a outage de provider; configurável via env, não hardcoded.
  12. status='Complete': receita realizada, exclui Cancelled/Returned. Métrica de negócio, não bruta.
  13. COUNT(DISTINCT order_id): evita inflação por order_items ter N linhas por pedido. Item técnico que separa SQL júnior de sênior.
  14. CAST AS NUMERIC: precisão decimal em receita; float acumula erro binário.
  15. Limitações explícitas no README: sem ROAS, sem UI, sem persistência durável — sinal de maturidade de escopo, não de incapacidade.

Como justificar qualquer escolha — framework

Padrão de resposta: [escolha] porque [requisito do case] exige [propriedade]. Alternativa [X] foi descartada porque [custo concreto]. Em [contexto Y], inverteria.

Mostre que conhece a alternativa, descartou conscientemente e sabe quando inverteria. Júnior responde "porque é bom"; sênior responde com trade-off.

[Geral] Explique a arquitetura em 2 minutos.

TL;DR: FastAPI → router determinístico → agent LLM com tools → tool_executor → BigQuery → síntese pt-BR. Multi-turn via thread_id + MemorySaver.

  • FastAPI recebe POST /query com pergunta + thread_id opcional.

  • Se short-circuit (recusa ou clarificação), responde direto sem LLM.
  • Senão, agent_node chama LLM com tools bound via tool_choice="auto".
  • LLM retorna AIMessage.tool_calls → tool_executor resolve por nome, valida args (Pydantic), executa SQL parametrizada no BQ.
  • ToolMessage volta para o agent que sintetiza resposta final em pt-BR.
  • Loop guard de 3 iterações previne runaway.
  • Multi-turn: MemorySaver indexa por thread_id.

Armadilha: não soletre "ReAct" sem necessidade. Foco na separação prompt × execução, é o que o critério "arquitetura" pondera alto.

[Sr DS] Como você sabe que a receita está correta?

TL;DR: 4 garantias compostas — COUNT(DISTINCT order_id), status='Complete', CAST AS NUMERIC, INNER JOIN coerente com a pergunta.

  • order_items tem N linhas por pedido. Sem DISTINCT, um pedido com 3 itens conta 3 vezes em total_orders.
  • SUM(sale_price) com FLOAT64 acumula erro binário. CAST AS NUMERIC garante precisão decimal.
  • Receita "realizada" exclui Cancelled/Returned via status='Complete'.
  • INNER JOIN porque só usuários que compraram contam para receita. LEFT JOIN distorceria com NULLs.
  • Validável: poetry run pytest -m live roda contra o BQ real.

Armadilha: se ele pedir conversion rate, mude para LEFT JOIN. É o sinal de que você entende o porquê do INNER, não copia padrão.

[Sr DS] Por que NUMERIC em vez de FLOAT64?

TL;DR: NUMERIC é decimal de precisão fixa (38 dígitos, 9 decimais). FLOAT64 é IEEE 754 binário — acumula erro em somas.

  • 0.1 + 0.2 em float = 0.30000000000000004.
  • Em 100k pedidos somados, erro vira centavos errados visíveis.
  • NUMERIC tem custo de storage maior, mas para colunas de dinheiro o trade-off é óbvio.
  • BigQuery também oferece BIGNUMERIC para 76 dígitos — overkill aqui.
[Geral] Tool calling vs LLM gerando SQL — por que tool calling?

TL;DR: segurança, custo, auditoria, testabilidade.

  • Segurança: queries fixas eliminam injection. SQL gerada por LLM precisaria sanitizar e validar AST.
  • Custo BQ: queries auditadas escaneiam só o necessário. LLM tende a SELECT *.
  • Custo LLM: tokens para "explique a query" são caros e redundantes.
  • Determinismo: testes cobrem SQL, não o output do LLM.
  • Schema garantido: args validados via Pydantic antes da execução.

Armadilha: não venda como "evita alucinação 100%" — o LLM ainda pode chamar tool errada. Venda como redução de superfície de risco.

[ETI] Como escalaria isso para produção?

TL;DR: durable checkpointer + cache + slot reservation + observability + deploy gerenciado.

  • Checkpointer: trocar MemorySaver por PostgresSaver ou RedisSaver. Threads sobrevivem deploy.
  • Cache: queries idênticas (mesma start_date/end_date/traffic_source) podem cachear no Redis com TTL.
  • BQ: slot reservation para custo fixo previsível; ou continuar pay-per-byte com budget alerts.
  • Deploy: Cloud Run (stateless) + Cloud SQL/Memorystore para o checkpointer. Cloud Functions se for low-traffic.
  • Observability: Langfuse para traces de LLM, OpenTelemetry para o resto, BQ jobs com labels.
  • Async/fila: queries longas viram task assíncrona; cliente consulta status. Evita timeout HTTP.
  • Rate limit: por thread_id ou IP para conter abuso de tokens.
  • Secrets: Secret Manager em vez de .env; rotação automática.
[ETI] Como você protege contra SQL injection?

18.1 · Agente custom vs create_react_agent

O que é ReAct. Padrão "Reasoning + Acting": LLM alterna Thought (raciocínio) → Action (chamar tool com args) → Observation (lê resultado) → repete até resposta final. Hoje "ReAct agent" virou sinônimo do loop LLM → tool → LLM → tool → ... → final, com tool calls em JSON estruturado em vez de texto Thought/Action.

Onde aparece em produção. Claude Code, Cursor, Codex/Copilot Workspace, Aider, Cline, ChatGPT (Python sandbox/browse), Claude com tools/MCP, Devin, OpenHands — todos rodam ReAct no núcleo. Diferenças não estão no padrão, mas em tool set, gestão de memória/contexto, layer de planning opcional e orquestração multi-agente. Variantes acadêmicas (Reflexion, Tree of Thoughts, LATS) existem mas produção comercial usa ReAct puro ~95% das vezes.

O que o agente custom deste projeto adiciona vs create_react_agent (prebuilt do LangGraph):

  • Prompt ramificado por intent (strategy_follow_up tem system prompt diferente de diagnostic_follow_up).
  • Contagem manual de iterações no turn corrente → guard _MAX_AGENT_ITERATIONS=3 com mensagem amigável.
  • Janela de histórico via turn_start_index (envia só o turn atual ao LLM, não o thread inteiro).
  • Injeção de router_guidance direto no system prompt (LLM sabe que router já validou datas/canal).
  • Observabilidade por node individual (cada node é função Python, traceável separadamente).

Senioridade. Não é "consegui fazer short-circuit" — é trade-off consciente. Júnior pega create_react_agent porque é o que o tutorial mostra; sênior abre mão da conveniência por controle e justifica.

18.2 · Por que router determinístico em vez de LLM responder "fora de escopo"

LLM sabe recusar, mas recusa via LLM custa em 4 dimensões:

  1. Custo. Toda pergunta absurda ("ROAS do Facebook?") consome tokens input+output. Em escala = dinheiro real.
  2. Latência. 1–3 s por chamada LLM. Router responde em microssegundos.
  3. Determinismo. LLM pode às vezes aceitar fora de escopo (alucinação), inventar ROAS com dados que não tem. Router nunca erra — é regex.
  4. Auditoria. OUT_OF_SCOPE_MESSAGE é constante, não varia por temperatura. Texto previsível para suporte.

Resumo: router economiza tokens e garante recusa confiável.

18.3 · Command(goto=...) dinâmico e a constante END

Pergunta natural: por que graph.add_edge("preprocess", END) não aparece? Resposta: o destino é decidido em runtime pelo próprio node, via Command.

if _router_decision_short_circuits(router_decision):
    return Command(update=state_update, goto="__end__")
return Command(update=state_update, goto="agent")
  • "__end__" é o sentinel string equivalente à constante END importada de langgraph.graph.
  • Quando node retorna Command(goto=X), LangGraph usa X — add_edge estático fica redundante para esse node.
  • Para o Studio/visualização desenhar a aresta tracejada, declare o tipo de retorno como Command[Literal["agent", "__end__"]] ou use add_conditional_edges. Sem isso, funciona mas a visualização perde a aresta.
  • "Pode não existir caminho pro fim em runtime?" Não — é como if/else em código normal: os dois branches existem no binário, só um executa por invocação. tool_executor → agent é estático porque sempre volta, sem decisão.

18.5 · Justificativa pronta para a entrevista (router)

Resposta-modelo: "Considerei LLM-based router. Descartei por 4 razões alinhadas ao case:

  1. Escopo fechado — o case define 3 tabelas e 2 perguntas-tipo. O espaço de intents cabe em regex. ML resolve cauda longa que aqui não existe.
  2. Critério 'fora de escopo' — o case avalia recusa explícita. Router determinístico recusa CAC/ROAS com 100% de reliability. LLM router daria ~95%, com 5% inventando.
  3. Custo previsível — toda request já paga LLM no agent. Adicionar LLM no router dobra custo sem ganho de qualidade aqui.
  4. Testabilidade — pytest cobre router. LLM router exigiria eval suite, fora do escopo do MVP.

Em domínio aberto (suporte ao cliente, copiloto genérico), inverteria — LLM router ganha. Híbrido seria o próximo passo: regex pega óbvios, LLM resolve ambíguos."

18.6 · Como o router classifica (ele não é um node)

Router é função pura build_router_decision(question: str) → RouterDecision chamada dentro do preprocess_node. Não há node "router" no grafo.

def build_router_decision(question: str) -> RouterDecision:
    norm = _normalize(question)              # NFKD + lower + strip acentos
    tokens = set(norm.split())

    if not norm:
        return RouterDecision(intent="empty", short_circuit=True, ...)

    if tokens & UNSUPPORTED_METRIC_TOKENS:    # CAC, ROAS, CTR, ...
        return RouterDecision(intent="out_of_scope", short_circuit=True, ...)

    start, end = _parse_dates(norm)           # regex (5 formatos)
    source = _detect_traffic_source(norm)     # dict de aliases

    if tokens & REVENUE_TOKENS:
        intent = "channel_performance"
    elif tokens & VOLUME_TOKENS:
        intent = "traffic_volume"
    elif tokens & STRATEGY_TOKENS and has_prev_context:
        intent = "strategy_follow_up"
    else:
        intent = "ambiguous_analytics"

    if intent in ("traffic_volume","channel_performance") and not (start and end):
        return RouterDecision(intent=intent, short_circuit=True,
                              response_message=CLARIFY_DATES_MESSAGE)

    return RouterDecision(intent=intent, traffic_source=source,
                          start_date=start, end_date=end, short_circuit=False)

Pipeline = monte de if/else + regex + lookups em frozenset. Determinístico de ponta a ponta.

18.8 · preprocess_node — o que é e por que é determinístico

  • Função Python (não LLM) que roda antes do agente.
  • Pega question do state.
  • Chama build_router_decision(question) (puro Python).
  • Atualiza state com router_decision, resolved_question, turn_start_index.
  • Se router pediu short-circuit → Command(goto="__end__", update={final_answer: ...}).
  • Senão → Command(goto="agent", update=...).

É a "porta de entrada barata" antes de gastar tokens. Determinístico porque não há LLM dentro — mesma pergunta gera mesma decisão sempre.

18.11 · Justificativas de stack — frases prontas

  • LangGraph > LangChain AgentExecutor: "LangGraph expõe nodes/edges/state explícitos. Permite preprocess determinístico, guard de iteração e prompts ramificados por intent. AgentExecutor esconde o loop ReAct numa classe — sem hook fácil para short-circuit."
  • FastAPI > Flask: "Pydantic v2 nativo, ASGI async, OpenAPI auto. Schemas das tools (que o LLM vê) reutilizam os mesmos modelos da API."
  • MemorySaver: "Case não exige durabilidade. MemorySaver = zero infra, suficiente para o MVP. Próximo passo declarado: Redis/Postgres."
  • Tool calling > LLM gera SQL: "Segurança (sem injection), custo BQ previsível (bytes scanned controlados), determinismo (SQL testável), auditável."
  • NUMERIC > FLOAT64: "Dinheiro. FLOAT64 = IEEE 754 binário, acumula erro em soma de N pedidos. NUMERIC = decimal precisão fixa."
  • Fallback de provider (with_fallbacks): "Resiliência sem code change. Se OpenAI cair, request continua via Anthropic."

Regra: sempre mostre que conhece a alternativa, descartou conscientemente e sabe quando inverteria a decisão.

18.12 · Eval suite — o que é

Conjunto de testes para LLM/agente. Análogo a test suite de código, mas para comportamento não-determinístico.

  • Dataset: pares (input, expected_output) ou (input, rubric). Ex: pergunta → intent esperada, tool esperada, métricas no resultado.
  • Runner: executa agente em cada caso, coleta resposta.
  • Judge: assertiva exata (intent correto) ou LLM-as-judge (resposta cobre os pontos?).
  • Métricas: accuracy, tool selection rate, hallucination rate, latência, custo.
  • Regression gate: CI roda eval em cada PR. Bloqueia merge se cair abaixo de threshold.

18.13 · Cache LRU — funcionamento

LRU = Least Recently Used. Cache de tamanho fixo que descarta o item acessado há mais tempo quando enche.

  • Estrutura: hash map + doubly-linked list (ou OrderedDict). Operações O(1).
  • get(key): se existe, move pra frente da lista, retorna valor.
  • put(key, val): insere na frente. Se passou de maxsize, remove o último.
  • No projeto: @lru_cache em get_persistent_analytics_graph() (singleton do grafo compilado).
from functools import lru_cache

@lru_cache(maxsize=128)
def cached_query(tool_name: str, source: str, start: str, end: str): ...

Aplicação futura em queries BQ: key = (tool_name, traffic_source, start_date, end_date), value = resultado. Hit economiza scan BQ (latência + custo). Tradeoff: stale data. Aceitável para ranges fechados no passado, arriscado para "hoje" (precisa TTL adicional).
